# V0.2 — Assimilation Inbox

V0.2 inserts a review layer between agent capture and durable memory. Instead of writing straight into the canonical store, agents propose candidates (`cand_<ULID>`). Candidates queue in an inbox, invisible to recall, until a human reviews them — approving to promote into a real `mem_<ULID>`, or rejecting with a reason. This keeps the memory store trustworthy: every recalled memory has been seen and endorsed by a human, not just emitted by an LLM.

The change that matters most for day-to-day use: `vestige-auto-memorise` now proposes candidates rather than writing durable memories. Explicit capture commands (`vestige-record-decision`, `vestige-record-note`, etc.) still write directly — those fire on deliberate user intent and need no gating.

---

## The flow

```
agent observes something
    |
    v
vestige candidate add          (or vestige_propose_candidate via MCP)
    |
    v
candidate_memories  (status=pending — invisible to search/recall/context)
    |
    v
  vestige inbox                 # list pending candidates
  vestige inbox show <cand_id>  # inspect one
    |
    +------- vestige approve <cand_id> ---------> mem_<ULID>  (normal memory, recalled immediately)
    |
    +------- vestige reject <cand_id> --reason …  (audit only — never recalled)
```

Approved candidates carry full provenance: the new `mem_<ULID>` gets a `memory_sources` row with `source_type='candidate'` and `source_ref=<cand_id>` for the reverse link; `candidate_memories.approved_memory_id` is the forward link. The candidate's original source rows are also copied to the promoted memory.

## Commands

### `vestige candidate add`

Propose a new candidate to the assimilation inbox.

```
vestige candidate add --type <TYPE> --body <BODY> [options]
```

| Flag | Required | Description |
|------|----------|-------------|
| `--type` | yes | `decision` \| `note` \| `observation` \| `preference` \| `open_question` \| `project_summary` |
| `--body` | yes | Full body of the candidate. |
| `--title` | no | Title override (derived automatically if omitted). |
| `--rationale` | no | Why this is worth recording. |
| `--importance` | no | Signal strength [0.0–1.0]. Default: `0.5`. |
| `--confidence` | no | Agent confidence [0.0–1.0]. Default: `0.5`. |
| `--source-type` | no | Source category (`file`, `url`, `clipboard`, …). Enables source attachment. |
| `--source-ref` | no | Source locator (file path, URL, …). Requires `--source-type`. |
| `--source-content` | no | Source content snippet. Capped at 2 KiB (UTF-8 boundary). Requires `--source-type`. |
| `--duplicate-of-memory` | no | `mem_<ULID>` this candidate duplicates. |
| `--duplicate-of-candidate` | no | `cand_<ULID>` this candidate duplicates. |
| `--json` | no | JSON output. |

```bash
vestige candidate add \
  --type decision \
  --body "Prefer ULID over UUID for all entity IDs — lexicographic sort aligns with created_at." \
  --rationale "Keeps queries ordered without an extra timestamp column." \
  --importance 0.8 \
  --source-type file \
  --source-ref "crates/vestige-core/src/ids.rs"
```

Output includes dedup hints: `similar_memories` (active memories that may already cover this) and `similar_candidates` (pending candidates that may be near-duplicates). Use these to decide whether to flag `--duplicate-of-memory` or `--duplicate-of-candidate`.

### `vestige inbox`

List pending candidates.

```
vestige inbox [options]
```

| Flag | Description |
|------|-------------|
| `--limit <N>` | Maximum rows to return. |
| `--type <TYPE>` | Filter by proposed type. |
| `--include-rejected` | Show rejected candidates alongside pending. |
| `--json` | JSON output (`{ candidates: [...] }`). |

```bash
vestige inbox
vestige inbox --type decision --json
vestige inbox --include-rejected --limit 20
```

---

### `vestige inbox show <id>`

Inspect a single candidate at full fidelity: body, rationale, sources, duplicate links, and next-action hints.

```
vestige inbox show <cand_id> [--json]
```

```bash
vestige inbox show cand_01JVXXXXXXXXXXXXXXXXXX
vestige inbox show cand_01JVXXXXXXXXXXXXXXXXXX --json
```

---

### `vestige approve <id>`

Promote a pending candidate to a durable memory.

```
vestige approve <cand_id> [options]
```

| Flag | Description |
|------|-------------|
| `--type <TYPE>` | Override the proposed memory type before promotion. |
| `--body <BODY>` | Override the candidate body before promotion. |
| `--importance <F>` | Override importance [0.0–1.0]. |
| `--json` | JSON output (`{ candidate_id, memory_id, status: "approved" }`). |

```bash
vestige approve cand_01JVXXXXXXXXXXXXXXXXXX
vestige approve cand_01JVXXXXXXXXXXXXXXXXXX --type note --json
```

The promoted `mem_<ULID>` is visible in `vestige search` / `vestige recall` / `vestige context` immediately.

---

### `vestige reject <id>`

Dismiss a pending candidate. Rejected candidates are retained for audit but never recalled.

```
vestige reject <cand_id> [options]
```

| Flag | Description |
|------|-------------|
| `--reason <REASON>` | `duplicate` \| `wrong` \| `not_durable` \| `too_noisy` \| `stale` \| any free-form string (stored as `other:<string>`). Defaults to `other:unspecified`. |
| `--duplicate-of <MEM_ID>` | `mem_<ULID>` this duplicates. Pair with `--reason duplicate`. |
| `--note <TEXT>` | Optional reviewer note. |
| `--json` | JSON output (`{ candidate_id, status, reason, duplicate_of }`). |

```bash
vestige reject cand_01JVXXXXXXXXXXXXXXXXXX --reason not_durable
vestige reject cand_01JVXXXXXXXXXXXXXXXXXX --reason duplicate --duplicate-of mem_01JVXXXXXXXXXXXXXXXXXX
vestige reject cand_01JVXXXXXXXXXXXXXXXXXX --reason wrong --note "Misread the type signature."
```

## MCP tools

Three tools are added in V0.2. All are read-write-gated by config and the global `--read-only` flag.

### `vestige_propose_candidate`

Propose a candidate for the inbox. Gated by `mcp.allow_propose_candidate` (default `true`) and `--read-only`.

**Parameters:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `type` | string | yes | Memory type (`decision`, `observation`, `note`, `preference`, `question`, `summary`). |
| `body` | string | yes | Full body text. |
| `title` | string | no | Title override. |
| `rationale` | string | no | Why this is worth recording. |
| `importance` | float | no | Default `0.5`. |
| `confidence` | float | no | Default `0.8`. |
| `source` | object | no | `{ type: string, ref?: string, content?: string }` |

**Response:** `{ candidate_id, status, similar_memories: [...], similar_candidates: [...] }`

```json
{
  "type": "decision",
  "body": "Always write migrations as immutable append-only files.",
  "rationale": "Old DBs in ~/.vestige/projects/ cannot re-run mutated migrations.",
  "importance": 0.9,
  "confidence": 0.95
}
```

---

### `vestige_list_candidates`

List candidates in the assimilation inbox. Read-only; no capability gate.

**Parameters:**

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `status` | string | `"pending"` | `pending` \| `approved` \| `rejected` \| `superseded` |
| `type` | string | — | Filter by memory type. |
| `limit` | integer | `50` | Maximum rows returned. |
| `include_rejected` | boolean | `false` | Return both pending and rejected rows. |

**Response:** `{ candidates: [ { id, type, status, title, one_liner, confidence, importance, created_at }, … ] }`

```json
{ "status": "pending", "limit": 10 }
```

---

### `vestige_get_candidate`

Fetch a single candidate at full fidelity, including sources, duplicate links, and approval/rejection metadata. Verifies project scope.

**Parameters:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `candidate_id` | string | yes | `cand_<ULID>` |

**Response:** Full `Candidate` object — all fields including `full_body`, `rationale`, `sources`, `duplicate_of_memory_id`, `approved_memory_id`, `rejection_reason`, timestamps.

```json
{ "candidate_id": "cand_01JVXXXXXXXXXXXXXXXXXX" }
```

### What is NOT in V0.2

`vestige_approve_candidate` and `vestige_reject_candidate` are **deliberately absent**. Approval is CLI-only until the review policy is proven and config semantics are settled. The config flags `mcp.allow_candidate_approval` / `mcp.allow_candidate_rejection` exist for forward-compatibility but have no tool surface yet.

---

## Provenance

Every approved memory carries a `memory_sources` row:
- `source_type = 'candidate'`, `source_ref = <cand_id>` — reverse link from memory back to candidate.
- `candidate_memories.approved_memory_id = <mem_id>` — forward link from candidate to memory.

If the candidate itself had source rows (file, URL, etc.), those are copied to the promoted memory as well. This means the full chain — from observed artifact through candidate through memory — is traceable in both directions with no extra joins.

---

## Recall isolation

Pending and rejected candidates are stored in a separate `candidate_memories` table. They are **never reachable** from:

- `vestige search` / `vestige recall` — queries only `memories` + `memory_fts`.
- `vestige context` — builds the pack from `list_memories` with active-only filters.
- `vestige_search` / `vestige_get_project_context` MCP tools — same engine paths.

Approved candidates appear in recall **immediately** after promotion — no re-index required.

---

## Skill behaviour change

`vestige-auto-memorise` now routes all proactive capture through `vestige candidate add` instead of the direct `vestige decision add` / `vestige note add` / etc. commands. The surface line changes from `Recorded decision mem_…` to `Proposed decision cand_…`.

Explicit capture skills are unaffected:

| Skill | Behaviour |
|-------|-----------|
| `vestige-auto-memorise` | Writes candidates (V0.2 change). |
| `vestige-record-decision` | Still writes durable `mem_<ULID>` directly. |
| `vestige-record-note` | Still writes durable `mem_<ULID>` directly. |
| `vestige-record-preference` | Still writes durable `mem_<ULID>` directly. |
| `vestige-record-question` | Still writes durable `mem_<ULID>` directly. |

The distinction: `vestige-auto-memorise` fires on ambient detection (the agent decides something might be worth saving); the explicit capture skills fire on direct user intent ("remember this decision") and are trusted enough to bypass the inbox.

## Configuration

New optional sections in `.vestige/config.toml`:

```toml
[mcp]
allow_propose_candidate    = true   # agents can write to the inbox via MCP
allow_candidate_approval   = false  # no MCP approval tool in V0.2
allow_candidate_rejection  = false  # no MCP rejection tool in V0.2

[assimilation]
enabled         = true
default_capture = "candidate"  # candidate | memory
```

All fields are optional. Existing configs that omit these sections round-trip without changes — defaults are applied at runtime.

Setting `assimilation.enabled = false` causes `vestige candidate add` and `vestige_propose_candidate` to return `CANDIDATE_DISABLED`. Setting `default_capture = "memory"` is a future hook; in V0.2, only `vestige-auto-memorise` honours it.

## What is not in V0.2

- No daemon or background watcher.
- No GUI or TUI inbox view.
- No LLM summarisation of candidates.
- No `$EDITOR`-backed approval (inline flags only).
- No contradiction detection.
- No auto-approval policy (timer, confidence threshold, etc.).
- No embeddings on candidates — dedup is lexical-only until promotion.
- No cross-project inbox — candidates are scoped per project, same as memories.
- No `vestige search --include-candidates` flag (deferred to V0.3).

---

## Migration and upgrade notes

`migration 0004_candidates.sql` is applied automatically the first time any V0.2 command opens the store. No manual step is needed. Existing `.vestige/config.toml` files round-trip without changes; the new `[mcp]` and `[assimilation]` sections are optional and default-safe.

If you downgrade to a V0.1 binary after running V0.2, the `candidate_memories` table will remain in the DB but be ignored — no data loss.

Migrations are immutable. The `0004_candidates.sql` file must not be edited after it ships; add a new numbered file for any future schema change.
