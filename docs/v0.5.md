# V0.5 — Daemon Runtime

V0.5 adds an opt-in background process — `vestige daemon` — that keeps your project stores open and handles scheduled maintenance jobs so you never have to remember to run them yourself.

Nothing about the core memory model changes. The store schema, migration rules, and all MCP/CLI surfaces carry forward unchanged. V0.5 is an operations layer: it does work between commands so the work is already done when a command runs.

---

## What V0.5 ships

- **`vestige daemon` subcommand family** — nine subcommands: `start`, `stop`, `status`, `kick`, `log`, `install`, `uninstall`, `restart`, `doctor`.
- **Scheduled maintenance jobs** — embed sweep (10 min), trace prune (24 h), candidate TTL (1 h, off by default) — running on tokio intervals.
- **Control socket** — `~/.vestige/daemon.sock`, a Unix-domain socket accepting JSON-RPC 2.0 for mutations (`kick`, `register_project`, `reload_config`); reads go through the status file.
- **Four IPC methods** — `daemon.status`, `daemon.kick`, `daemon.register_project`, `daemon.reload_config` (live cadence reload without restart).
- **macOS LaunchAgent install** — `vestige daemon install` writes and loads `~/Library/LaunchAgents/com.vestige.daemon.plist`. The daemon starts at login and is restarted by launchd within seconds of a crash.
- **Daemon surface in `vestige status`** — the existing `vestige status` command gains a one-line `daemon: running pid=… last_embed=…` block when a daemon is alive.
- **Per-project provider selection** — each worker reads the project's `[embeddings]` config at spawn time so daemon embeddings match `vestige embed --all` for the same project.
- **`vestige init` registers with the daemon** — a fresh `init` fires a `daemon.register_project` ping so new projects appear in the supervised list immediately, without restarting the daemon.

---

## The 30-second tour

```bash
vestige daemon install        # writes plist + launchctl load -w
vestige daemon status         # text/JSON snapshot
vestige daemon doctor         # 8-check health diagnostic — your first stop when something's off
vestige daemon kick embed     # one-off embed sweep
vestige daemon log -f         # follow the rolling daemon log
vestige daemon restart        # bounce via launchctl kickstart
vestige daemon uninstall      # tear down
```

The daemon runs in the background from this point. Embeddings stay current. No further commands are required.

## Setup

The daemon needs a Vestige project in the current repo (run `vestige init` once if you haven't already). It then discovers all project databases under `~/.vestige/projects/*/` automatically — no extra registration step for existing projects.

On first run, `vestige daemon install`:

1. Renders `~/Library/LaunchAgents/com.vestige.daemon.plist` using the resolved path to the `vestige` binary.
2. Calls `launchctl load -w` to register and start the agent immediately.
3. Prints confirmation including the plist path and the vestige binary it is wired to.

Use `--no-load` to write the plist without starting the daemon (useful in CI). Use `--force` to overwrite an existing plist. Use `--bin /path/to/vestige` to override the binary path (for Homebrew-managed installs that may have moved after an upgrade).

## What the daemon does

### Embed sweep

Runs every 600 seconds (10 minutes) by default. Calls the same `embed_all` function the CLI uses — it walks every memory representation for the project, finds any without embeddings, and generates them via the configured provider. On idle projects the query returns an empty set instantly; there is no busy-loop overhead.

The sweep means you can safely run `vestige search --mode semantic` on freshly recorded memories without manually running `vestige embed --all`. The daemon takes care of it.

Cadence controlled by `embed_sweep_interval_secs` in `[daemon]`.

### Trace prune

Runs every 86400 seconds (24 hours). Evicts old `query_events` rows that exceed the per-project FIFO cap (default 10000 traces) and then VACUUMs the database. This is the same eviction logic that inline CLI calls perform, but moved to a scheduled job so it does not add latency to individual commands.

Cadence controlled by `trace_prune_interval_secs` in `[daemon]`.

### Candidate TTL

Runs every 3600 seconds (1 hour) when `candidate_ttl_days` is greater than zero. Marks assimilation candidates older than that many days as rejected with `reason='expired'`. This is a soft operation — rejected candidates are never deleted and can be inspected in `vestige browse`.

Disabled by default (`candidate_ttl_days = 0`). Enable by adding `candidate_ttl_days = 14` (or any positive value) to `[daemon]` in `.vestige/config.toml`.

Cadence controlled by `candidate_ttl_sweep_interval_secs` in `[daemon]`.

## What it doesn't do

- **MCP-talks-to-daemon.** Agents communicate with the MCP server over stdio as before. The daemon has no MCP surface. Unscheduled backlog (V0.6 is Directives).
- **Cross-project federation.** Each worker holds exactly one project's Store. Project A cannot see project B. Deferred to V0.7.
- **File-system watching.** The daemon polls on a timer rather than watching `memory.sqlite` for changes. The 10-minute cadence is sufficient and avoids debounce complexity.
- **Linux/Windows.** macOS only in V0.5 (`vestige daemon install` exits with a clear error on other platforms). Linux systemd user-service support is unscheduled backlog; Windows later still.
- **REM consolidation jobs.** Memory rewriting and clustering shifted to V0.7 (REM = Review · Evaluate · Merge).
- **iOS.** iOS has no Unix sockets accessible across apps, no LaunchAgents equivalent, and no way to run a Rust process against a local `~/.vestige/` hierarchy. An iOS surface would require a server-hosted Vestige — that is V0.8+ and a different product shape.

## Architecture in one diagram

```
CLI/MCP (one-shot)  ──┐
                      ├──► SQLite (WAL + busy_timeout=5000ms)
vestige-daemon ───────┘
    │
    ├── scheduler (tokio intervals: embed/prune/ttl/status)
    ├── per-project worker thread (owns one Store, !Send)
    ├── status writer  →  ~/.vestige/daemon.status.json  (every 5s, atomic)
    └── control socket →  ~/.vestige/daemon.sock  (JSON-RPC 2.0)
```

The key source files in `crates/vestige-daemon/src/`:

- `lib.rs` — `run()` entry point: pidfile acquire, signal hook, task spawn.
- `lifecycle.rs` — pidfile locking with `fcntl` advisory lock; graceful SIGTERM/SIGINT drain.
- `scheduler.rs` — four tokio interval ticks (embed, prune, ttl, status); biased select exits cleanly on cancellation.
- `workers.rs` — per-project worker thread owning one `Store`; `mpsc` command channel for embed/prune/ttl/ping.
- `registry.rs` — `ProjectRegistry` scans `~/.vestige/projects/*/memory.sqlite` at startup and accepts `ensure_registered` calls from the IPC server for new projects.
- `ipc/server.rs` — Unix-domain socket accept loop; one request per connection.
- `ipc/methods.rs` — four JSON-RPC methods: `daemon.status`, `daemon.kick`, `daemon.register_project`, `daemon.reload_config`.
- `ipc/status_file.rs` — atomic write of `DaemonStatus` to `daemon.status.json` via write-to-`.tmp` + `rename`.
- `plist.rs` — LaunchAgent plist template rendering (`include_str!` at compile time).

## IPC methods

The control socket at `~/.vestige/daemon.sock` accepts one JSON-RPC 2.0 request per connection. The four methods:

### `daemon.status`

Returns the same snapshot written to `daemon.status.json`. Use this when you need a live read over the socket rather than a file stat.

```json
// Request
{"jsonrpc":"2.0","id":1,"method":"daemon.status","params":{}}

// Response (abbreviated)
{"jsonrpc":"2.0","id":1,"result":{"schema_version":1,"pid":12345,...}}
```

### `daemon.kick`

Runs a scheduled job immediately across all projects (or one specific project).

```json
// Request — kick embed sweep across all projects
{"jsonrpc":"2.0","id":1,"method":"daemon.kick","params":{"job":"embed"}}

// Request — kick ttl sweep for one project
{"jsonrpc":"2.0","id":1,"method":"daemon.kick","params":{"job":"ttl","project_id":"proj_vestige"}}

// Response
{"jsonrpc":"2.0","id":1,"result":{"queued":true,"queued_at":"2026-05-19T11:14:41Z","projects_queued":2}}
```

Valid `job` values: `embed`, `prune`, `ttl`.

### `daemon.register_project`

Upserts a project into the registry. Idempotent — returns `registered: false` if the project is already supervised.

```json
// Request
{"jsonrpc":"2.0","id":1,"method":"daemon.register_project",
 "params":{"project_id":"proj_vestige","project_name":"Vestige","repo_root":"/Users/conor/Development/Vestige"}}

// Response
{"jsonrpc":"2.0","id":1,"result":{"registered":true,"project_id":"proj_vestige"}}
```

### `daemon.reload_config`

Re-reads the daemon config from disk and pushes new cadences to the scheduler. The response field `applied_at: "next-tick"` reflects honest timing — the new values take effect on the next scheduler iteration, not mid-tick.

**What is reloaded live:** `embed_sweep_interval_secs`, `trace_prune_interval_secs`, `candidate_ttl_sweep_interval_secs`, `candidate_ttl_days`.

**What requires a restart:** provider changes (e.g. switching from `fake` to `fastembed`) — provider lifecycle includes model loading that cannot be hot-swapped safely.

```json
// Request
{"jsonrpc":"2.0","id":1,"method":"daemon.reload_config","params":{}}

// Response
{"jsonrpc":"2.0","id":1,"result":{
  "reloaded": true,
  "applied_at": "next-tick",
  "embed_sweep_interval_secs": 600,
  "trace_prune_interval_secs": 86400,
  "candidate_ttl_sweep_interval_secs": 3600,
  "candidate_ttl_days": 0,
  "note": "Provider changes require daemon restart — reload only updates job cadences and TTL config."
}}
```

## Config knobs

Add a `[daemon]` section to `.vestige/config.toml` to tune behaviour. All fields are optional; omitting the section uses the defaults shown.

```toml
[daemon]
enabled                        = false   # opt-in; true after `vestige daemon install`
embed_sweep_interval_secs      = 600     # 10 minutes
trace_prune_interval_secs      = 86400   # 24 hours
candidate_ttl_days             = 0       # 0 = disabled; set e.g. 14 to auto-expire
candidate_ttl_sweep_interval_secs = 3600 # 1 hour (only relevant if ttl_days > 0)
log_level                      = "info"  # error | warn | info | debug | trace

# Override paths — useful in tests or unusual installs.
# socket_path      = "~/.vestige/daemon.sock"
# status_file_path = "~/.vestige/daemon.status.json"
```

Environment overrides for test isolation: `VESTIGE_DAEMON_SOCK`, `VESTIGE_DAEMON_STATUS`, `VESTIGE_DAEMON_PID`, `VESTIGE_DAEMON_LOG`. These variables are consumed by test harnesses; normal users never need them.

## Inspecting the daemon

Three observability surfaces, from cheapest to most informative:

### `~/.vestige/daemon.status.json`

The daemon rewrites this file atomically every 5 seconds. Readers never see a partial or truncated file (write-to-`.tmp` + POSIX `rename`). This file exists only while the daemon is running; absence means the daemon is not up.

```json
{
  "schema_version": 1,
  "version": "0.5.0",
  "pid": 12345,
  "started_at": "2026-05-19T08:00:00Z",
  "uptime_secs": 3600,
  "projects": [
    {
      "project_id": "proj_vestige",
      "project_name": "Vestige",
      "repo_root": "/Users/conor/Development/Vestige",
      "last_embed_run": "2026-05-19T08:55:00Z",
      "last_prune_run": null,
      "last_ttl_run": null,
      "pending_embeds": 0
    }
  ],
  "next_jobs": [
    {"kind": "embed",         "at": "2026-05-19T11:24:41Z"},
    {"kind": "prune",         "at": "2026-05-20T11:14:41Z"},
    {"kind": "candidate_ttl", "at": "2026-05-19T12:14:41Z"}
  ]
}
```

`next_jobs` entries omit `project_id` when the job sweeps all projects — the field is `None` for sweep-all jobs (serialized with `skip_serializing_if = "Option::is_none"`) because the daemon kicks all registered projects in one tick rather than targeting one specifically.

### `~/.vestige/daemon.log`

Structured `tracing` output for the daemon process. Log level is controlled by `log_level` in `[daemon]` (default `info`) or overridden at runtime by `VESTIGE_LOG=debug`. The plist routes both stdout and stderr here.

```bash
vestige daemon log          # last 100 lines
vestige daemon log -n 500   # last 500 lines
vestige daemon log -f       # follow (tail -F under the hood)
```

### `vestige daemon status`

Reads `daemon.status.json` and formats it for humans or scripts. Works even when the control socket is down.

```bash
vestige daemon status           # human-readable one-liner + project list
vestige daemon status --json    # raw DaemonStatus JSON
vestige daemon status --watch   # refresh every 5s (Ctrl-C to stop)
```

Sample text output:

```
daemon: running  pid=12345  uptime=3600s  version=0.5.0
projects: 2
  proj_vestige (Vestige) — pending_embeds=0  last_embed=2026-05-19T08:55:00Z
  proj_my_app  (My App)  — pending_embeds=3  last_embed=2026-05-19T08:10:00Z
```

## `vestige daemon doctor`

`vestige daemon doctor` runs 8 independent checks and prints a structured `[OK]`/`[WARN]`/`[FAIL]` report. It is the first command to run whenever something looks wrong — it surfaces the most common problems (stale pidfile, path drift after a Homebrew upgrade, socket down) without requiring you to dig through logs manually.

```
vestige daemon doctor

[OK]    pidfile               pid=88379, process alive
[OK]    launchctl loaded      com.vestige.daemon listed
[OK]    plist                 ~/Library/LaunchAgents/com.vestige.daemon.plist, plutil OK
[OK]    plist bin path        plist and current binary both point at ~/.cargo/bin/vestige
[OK]    socket                ~/.vestige/daemon.sock reachable, replied in 5ms
[OK]    status file           ~/.vestige/daemon.status.json, age=2s
[OK]    projects open         10 found, 10 opened
[OK]    rolling log dir       most recent: daemon.log.2026-05-19 (age=2s)

overall: OK
```

**Check details:**

1. **pidfile** — `~/.vestige/daemon.pid` exists and the PID it contains is a live process (`kill -0`). FAIL if the file is missing or contains a PID with no matching process (stale after `SIGKILL`).

2. **launchctl loaded** — `com.vestige.daemon` appears in `launchctl list` output. WARN (not FAIL) if absent — the daemon may be running manually via `vestige daemon start` rather than as a LaunchAgent.

3. **plist** — `~/Library/LaunchAgents/com.vestige.daemon.plist` exists and passes `plutil -lint`. WARN if the plist is missing (run `vestige daemon install`); FAIL if it exists but is malformed.

4. **plist bin path** — the `ProgramArguments[0]` in the plist matches the current `vestige` binary path. WARN if they differ — common after a Homebrew upgrade moves the binary. Fix: `vestige daemon uninstall && vestige daemon install`.

5. **socket** — connects to `~/.vestige/daemon.sock` and sends a `daemon.status` ping; expects a reply within 2 seconds. FAIL if the socket is not reachable — the daemon is either down or the IPC task has not started yet.

6. **status file** — `~/.vestige/daemon.status.json` exists and its mtime is within 30 seconds. FAIL if absent; WARN if older than 30 seconds (the scheduler may be stuck).

7. **projects open** — every `~/.vestige/projects/*/memory.sqlite` opens cleanly via `Store::open`. FAIL if any database refuses to open. WARN if no project directories are found at all.

8. **rolling log dir** — `~/.vestige/logs/` exists and its most-recent file was written within 24 hours. WARN if the directory is absent or empty (run `vestige daemon start` to create it).

Exit code is **0** for all-OK or warn-only; **1** if any check is FAIL. Use `--json` for script-friendly output.

## `vestige daemon restart`

When the daemon is installed as a LaunchAgent, `vestige daemon restart` runs `launchctl kickstart -k gui/$UID/com.vestige.daemon` — the launchd-correct way to terminate and immediately respawn the process under KeepAlive. This avoids the `launchctl unload`/`launchctl load` dance, which can leave the agent in a disabled-override state.

If the LaunchAgent is not loaded (e.g. you started the daemon manually with `vestige daemon start`), restart bails with a clear error directing you to use `daemon stop` then `daemon start` to bounce it instead.

```bash
vestige daemon restart --json
```

```json
{
  "restarted": true,
  "launchctl_managed": true,
  "label": "com.vestige.daemon",
  "target": "gui/501/com.vestige.daemon"
}
```

## Per-project provider selection

Each worker thread reads its project's `.vestige/config.toml` `[embeddings]` block at spawn time and constructs the configured provider via `vestige_embed::build_provider`. If no `[embeddings]` block is present, the daemon falls back to the deterministic `FakeEmbeddingProvider`. This matches the behaviour of `vestige embed --all` — the daemon's embeddings will not drift from your manually-run embeddings in the same project.

Example `[embeddings]` blocks:

```toml
# Deterministic fake (default, no config needed)
[embeddings]
provider = "fake"
dimensions = 384

# On-device model via fastembed (compile with --features fastembed)
[embeddings]
provider = "fastembed"
model    = "BAAI/bge-small-en-v1.5"

# Ollama local server
[embeddings]
provider    = "ollama"
model       = "nomic-embed-text"
base_url    = "http://localhost:11434"
```

Provider changes (switching between fake/fastembed/ollama) require a daemon restart — provider lifecycle includes model loading that cannot be hot-swapped while workers are running. Cadence changes (`embed_sweep_interval_secs`, etc.) reload live via `daemon.reload_config`.

## `vestige init` registers with the running daemon

When you run `vestige init` in a new repo, the CLI fires a `daemon.register_project` RPC to `~/.vestige/daemon.sock` (500 ms timeout, fire-and-forget). If the daemon is running, the new project appears in its supervised list within seconds — no daemon restart needed. If the daemon is down, init exits silently with a debug log; the new project will be picked up the next time the daemon starts and scans `~/.vestige/projects/*/`.

This is always safe to run multiple times — `register_project` is idempotent and returns `registered: false` on a second call for the same project ID.

## Troubleshooting

When something looks wrong, start with `vestige daemon doctor`. It runs all 8 health checks in one pass and usually points directly at the problem — stale pidfile, binary path drift after a Homebrew upgrade, socket unreachable — faster than working through the FAQ entries below manually.

**"daemon already running pid=X"**

Another daemon instance is holding the pidfile lock. Stop it first: `vestige daemon stop`. If the process is gone but the pidfile remains (e.g. from a `SIGKILL`), remove it manually: `rm ~/.vestige/daemon.pid`.

**launchd unloaded the daemon / daemon not restarting after a crash**

Check the log: `vestige daemon log`. A common cause after a Homebrew upgrade is path drift — the plist still points to the old binary location. Fix it: `vestige daemon uninstall && vestige daemon install`.

**`vestige daemon status` says "not running" but `pgrep vestige` shows a process**

Stale pidfile from a `SIGKILL` — the daemon wrote its PID but could not clean up before dying. `vestige daemon start` reads the pidfile, checks whether that PID is still alive, and reclaims it if not. If `vestige daemon start` still complains, manually remove `~/.vestige/daemon.pid` and restart.

**`vestige daemon kick embed` says "could not reach daemon socket"**

The control socket is not up. Either the daemon is not running (`vestige daemon status` will confirm) or it started but the IPC task has not yet bound the socket (rare race on very slow systems). Start the daemon with `vestige daemon start` (or install it with `vestige daemon install`) and try again.

**embeddings not advancing despite the daemon running**

Check `pending_embeds` in `vestige daemon status`. If it is zero the daemon has no work to do. If it is non-zero and not moving, check `daemon.log` for errors from the embed sweep. The most common cause is a provider mismatch — the `[embeddings]` section in `.vestige/config.toml` names a provider that is not compiled in (e.g. `fastembed` without the `--features fastembed` build flag). The daemon logs a warning and falls back to `FakeEmbeddingProvider` in that case; fake embeddings are stored but won't produce meaningful semantic search results. Fix: rebuild with the correct feature flag and run `vestige daemon restart`.

## What's next (V0.6+)

**V0.5.1** shipped the macOS menu-bar Swift app (`Vestige.app`, `MenuBarExtra`) as a read-only window over the daemon status file. **V0.5.2** (next) adds menu-bar controls — kick / reload-config / pause from the menu, a `daemon.pause` IPC method, a persistent project workspace window, and easy boot (a `vestige ui` launcher plus `SMAppService` start-at-login, offered via an opt-in prompt on `vestige init` that is skipped in agent/CI contexts) (spec: `docs/prd/vestige_v_0_5_2_menubar_controls_prd.md`). **V0.6** is **Directives** — pluggable prompt blocks (`.vestige/directives.md`) plus project-scoped allow/deny rules injected into auto-memorise and heartbeat ingestion. The daemon's deeper integration story — MCP-talks-to-daemon (agents querying daemon health and triggering jobs over MCP) and Linux systemd user-service support — remains unscheduled backlog (see §19 below), not part of V0.6. The canonical, current roadmap ordering lives in `docs/src/data.js`.
