# V0.3 — Provenance and Receipts

V0.3 makes the memories Vestige already has **traceable, replayable, and trustworthy**. V0 proved explicit project memory. V0.1 added embeddings and hybrid recall. V0.2 added the assimilation inbox so capture became reviewable. V0.3 closes the loop by making every memory answerable to "where did this come from?" and every recall answerable to "what did the agent ask, and what did it get?".

```text
memory ──► candidate? ──► raw event ──► source snippet
   ▲                                          │
   └──── recalled by ◄──── query trace ◄──────┘
```

The data has been accumulating in `memory_events`, `memory_sources`, and the candidate audit trail since V0. V0.3 surfaces it.

All provenance commands are **project-scoped** — the same boundary as memories themselves. A `vestige why` or `vestige trace` in one repo cannot reach into another repo's store.

---

## New commands at a glance

| Command | What it does |
|---------|-------------|
| `vestige why <id>` | Templated provenance walk for a memory or candidate |
| `vestige sources <id>` | Raw typed source receipts for a memory or candidate |
| `vestige trace` | List recent recall traces |
| `vestige trace <trace_id>` | Full detail for one trace |
| `vestige trace replay <trace_id>` | Re-run a stored trace; diff against original |

Two MCP tools grow:

| Tool | What changed |
|------|-------------|
| `vestige_expand` | Accepts `depth="provenance"` for a structured walk |
| `vestige_trace` | New — `action=list\|show\|replay` |

## Setup

If you're working through this from scratch, follow the V0 / V0.1 / V0.2 setup first (one `vestige init`, a few memories, maybe a candidate or two). V0.3 adds no new capture commands — it interrogates what's already there.

```bash
cd ~/code/my-project
vestige init --name "My Project"

# Capture a memory directly:
vestige decision add "Prefer append-only migrations." \
  --rationale "Old DBs cannot re-run a mutated migration file."

# Or via the candidate inbox (V0.2 flow):
vestige candidate add \
  --type decision \
  --body "Use ULID over UUID for all entity IDs — lexicographic sort aligns with created_at." \
  --rationale "Keeps queries ordered without an extra timestamp column." \
  --importance 0.8

vestige inbox
# → cand_01JVXXXXXXXXXXXXXXXXXX  pending  decision  "Use ULID over UUID…"

vestige approve cand_01JVXXXXXXXXXXXXXXXXXX
# → Approved: promoted to mem_01JWXXXXXXXXXXXXXXXXXX

vestige status
# → Project: My Project (proj_my_project)
# → DB:      ~/.vestige/projects/proj_my_project/memory.sqlite
# → Memories: 2 active
```

Now you have at least one directly-recorded memory and one candidate-promoted memory. V0.3 gives you a receipt for both.

## `vestige why`

A recalled memory feels suspicious. You want to know how it got into the store.

```bash
vestige why mem_01JWXXXXXXXXXXXXXXXXXX
```

```text

  Title: Use ULID over UUID for all entity IDs
  Status: active  (since 2026-05-08)

  ◇ Proposed   2026-05-08 11:23:47  (evt_01JVXXXXXXXXXXXXXXXXXX)  source: agent_session
  ◇ Approved   2026-05-08 11:24:03  (evt_01JWXXXXXXXXXXXXXXXXXX)

Sources (2):
  ─ candidate      ref=cand_01JVXXXXXXXXXXXXXXXXXX   reverse-provenance link
  ─ agent_session  ref=current   "Keeps queries ordered without…"
                   (truncated: false; 0.3 KiB / 2 KiB)

Status history:
  2026-05-08 11:24  recorded
```

For a memory recorded directly (no candidate):

```bash
vestige why mem_01HVXXXXXXXXXXXXXXXXXX
```

```text
mem_01HVXXXXXXXXXXXXXXXXXX  decision  importance 0.5
  Title: Prefer append-only migrations
  Status: active  (since 2026-05-08)
```

`vestige why` also accepts `cand_<ULID>` to inspect a candidate that has not yet been approved. Soft-deleted memories are included — provenance must remain inspectable for forgotten memories so you can audit what was removed.

```bash
vestige why mem_01HVXXXXXXXXXXXXXXXXXX --json        # JSON output
vestige why mem_01HVXXXXXXXXXXXXXXXXXX --depth full  # inline source snippets
vestige why cand_01JVXXXXXXXXXXXXXXXXXX              # pre-approval candidate
```

## `vestige sources`

You want the raw evidence rows, not the narrative.

```bash
vestige sources mem_01JWXXXXXXXXXXXXXXXXXX
```

```text
mem_01JWXXXXXXXXXXXXXXXXXX  2 sources

src_01JWXXXXXXXXXXXXXXXXXX  candidate       cand_01JVXXXXXXXXXXXXXXXXXX
src_01JVXXXXXXXXXXXXXXXXXX  agent_session   current
                            "Use ULID over UUID for all entity IDs — lexicographic sort…"
                            (truncated: false; 0.3 KiB / 2 KiB)
```

Source kinds in V0.3:

| Kind | Meaning |
|------|---------|
| `file` | A file path on disk |
| `commit` | A git commit reference |
| `url` | A URL |
| `agent_session` | The active agent session |
| `mcp_call` | An MCP tool invocation |
| `candidate` | Reverse-provenance link on approval (V0.2) |
| `manual` | Recorded with no `--source` |
| `trace` | Forward-link to a query trace (reserved, V0.4+) |

Filter by kind, or target a candidate:

```bash
vestige sources mem_01JWXXXXXXXXXXXXXXXXXX --kind agent_session
vestige sources cand_01JVXXXXXXXXXXXXXXXXXX          # candidate sources
vestige sources mem_01JWXXXXXXXXXXXXXXXXXX --json    # JSON output
```

## `vestige trace`

Every `vestige search`, `vestige recall`, `vestige expand`, and `vestige context` call now writes one row to a dedicated `query_events` table. This happens automatically — no configuration required with the defaults.

List recent traces:

```bash
vestige trace
```

```text
Recent query traces (last 10):

trace_01JWXXXXXXXXXXXXXXXXXX  2026-05-08 14:02:11  search   hybrid   "ULID migration ordering"   2 results  43ms  caller=mcp
trace_01JVXXXXXXXXXXXXXXXXXX  2026-05-08 14:01:58  expand   —        mem_01JWXX… (depth=summary)  1 result   3ms   caller=mcp
trace_01JUXXXXXXXXXXXXXXXXXX  2026-05-08 13:58:42  context  —        (project bootstrap)          3 items    12ms  caller=cli
```

Inspect one trace in full:

```bash
vestige trace trace_01JWXXXXXXXXXXXXXXXXXX
```

```text
trace_01JWXXXXXXXXXXXXXXXXXX   search · hybrid    caller=mcp
Time:           2026-05-08 14:02:11 (43ms)
Query:          "ULID migration ordering"
Mode requested: hybrid     resolved: hybrid
Provider:       fake       model: deterministic-sha256
Limit: 8        Type filter: —

Results (2):
  1. mem_01JWXXXXXXXXXXXXXXXXXX  0.83  "Use ULID over UUID for all entity IDs"
  2. mem_01HVXXXXXXXXXXXXXXXXXX  0.61  "Prefer append-only migrations"

Score parts: lexical+vector merged via reciprocal rank fusion.
```

Filter the list:

```bash
vestige trace --limit 50
vestige trace --kind search
vestige trace --caller mcp
vestige trace --since 2026-05-08
vestige trace trace_01JWXXXXXXXXXXXXXXXXXX --json
```

## `vestige trace replay`

Replay re-runs a stored trace against the current store and current provider. It answers "if I ran that same query now, would I get the same memories?"

```bash
vestige trace replay trace_01JWXXXXXXXXXXXXXXXXXX
```

```text
Replaying trace_01JWXXXXXXXXXXXXXXXXXX

Original (2026-05-08 14:02:11):
  1. mem_01JWXXXXXXXXXXXXXXXXXX  0.83
  2. mem_01HVXXXXXXXXXXXXXXXXXX  0.61

Now (2026-05-08 14:30:00):
  1. mem_01JWXXXXXXXXXXXXXXXXXX  0.83   (unchanged)
  2. mem_01HVXXXXXXXXXXXXXXXXXX  0.61   (unchanged)

Provider: fake deterministic-sha256  (matches original)
Corpus drift: 0 memories since original.
```

If the corpus has changed since the original trace, drift shows explicitly:

```bash
vestige trace replay trace_01JWXXXXXXXXXXXXXXXXXX --json
```

```json
{
  "trace_id": "trace_01JWXXXXXXXXXXXXXXXXXX",
  "original": { "result_ids": ["mem_01JW…", "mem_01HV…"], "scores": [0.83, 0.61] },
  "current":  { "result_ids": ["mem_01JW…", "mem_01HV…", "mem_01KA…"], "scores": [0.83, 0.61, 0.55] },
  "diff": {
    "added":   ["mem_01KA…"],
    "removed": [],
    "score_changes": []
  },
  "provider_match": true,
  "corpus_drift": 1
}
```

Replay writes its own `query_events` row tagged `caller=mcp` (or `cli`) and `replay_of: trace_01JW…` in `params_json` so the chain is inspectable. It never mutates the original trace or any memory.

## MCP: `vestige_expand` with `depth="provenance"`

V0.3 extends `vestige_expand` rather than adding a new tool. Agents that already call `vestige_expand` for content now have a provenance path at the same address.

From the registered description:

> Expand a memory at a chosen representation depth: `one_liner | summary | compressed | full | provenance`. Depths `one_liner`–`full` return the title, type, depth, and content. `depth=provenance` returns the full provenance walk (events, candidate back-reference, source receipts) per PRD §10.2.

Request:

```json
{
  "memory_id": "mem_01JWXXXXXXXXXXXXXXXXXX",
  "depth": "provenance"
}
```

Response:

```json
{
  "memory_id": "mem_01JWXXXXXXXXXXXXXXXXXX",
  "type": "decision",
  "status": "active",
  "provenance": {
    "events": [
      { "event_id": "evt_01JWXXXXXXXXXXXXXXXXXX", "type": "memory.recorded", "at": "2026-05-08T11:24:03Z" }
    ],
    "candidate": {
      "candidate_id": "cand_01JVXXXXXXXXXXXXXXXXXX",
      "events": [
        { "event_id": "evt_01JVXXXXXXXXXXXXXXXXXX", "type": "candidate.proposed", "at": "2026-05-08T11:23:47Z" },
        { "event_id": "evt_01JWXXXXXXXXXXXXXXXXXX", "type": "candidate.approved", "at": "2026-05-08T11:24:03Z" }
      ]
    },
    "sources": [
      { "source_id": "src_01JWXXXXXXXXXXXXXXXXXX", "kind": "candidate", "ref": "cand_01JVXXXXXXXXXXXXXXXXXX" },
      { "source_id": "src_01JVXXXXXXXXXXXXXXXXXX", "kind": "agent_session", "ref": "current",
        "content_preview": "Use ULID over UUID for all entity IDs…", "truncated": false }

  }
}
```

`depth=provenance` is additive — the existing `one_liner`, `summary`, `compressed`, and `full` depths are unchanged.

## MCP: `vestige_trace`

From the registered description:

> Inspect and replay query traces. `action=list` returns recent traces (filterable by `kind`/`caller`/`since`, default limit 10). `action=show` returns full detail for a given `trace_id`. `action=replay` re-runs a stored trace against the current store, diffs the results, and writes a new `query_events` row tagged `caller=mcp` with `replay_of` in `params_json`.

List recent traces:

```json
{ "action": "list", "limit": 20, "kind": "search" }
```

Show one trace:

```json
{ "action": "show", "trace_id": "trace_01JWXXXXXXXXXXXXXXXXXX" }
```

Replay:

```json
{ "action": "replay", "trace_id": "trace_01JWXXXXXXXXXXXXXXXXXX" }
```

MCP-originated recall calls (`vestige_search`, `vestige_expand`, `vestige_get_project_context`, `vestige_bootstrap`) are automatically tagged `caller=mcp` in `query_events`. CLI calls are tagged `caller=cli`. No extra tool calls needed — tracing is transparent.

## `[traces]` configuration

All trace defaults are production-ready out of the box. Add the `[traces]` section to `.vestige/config.toml` only if you need to tune the behaviour.

```toml
[traces]
enabled                   = true    # false skips all trace writes; existing rows stay readable
max_per_project           = 10000   # FIFO cap: oldest rows evicted beyond this limit
truncate_query_text_bytes = 1024    # max bytes stored in query_text; truncated at UTF-8 boundary
trace_caller_cli          = true    # false: CLI recall calls are not traced
trace_caller_mcp          = true    # false: MCP recall calls are not traced
```

These are the exact field names from `TracesConfig`. Omitting the section entirely applies the same defaults. Omitting individual keys within the section applies their defaults individually — a partial block is safe.

`enabled = false` disables trace *writes* entirely. Existing `query_events` rows remain readable via `vestige trace` and replay.

`trace_caller_cli` and `trace_caller_mcp` let you disable tracing per surface — for example, disable MCP tracing on a busy agent while keeping CLI audit intact.

## Tracing behaviour

- **Mutations are not traced** — `vestige record`, `vestige forget`, `vestige restore`, `vestige approve`, `vestige reject` continue to write to `memory_events` only.
- **Traces are never recalled** — `vestige search`, `vestige recall`, and `vestige context` ignore `query_events` entirely.
- **Replay writes its own trace** — tagged with `caller=mcp|cli` and `replay_of: trace_<ULID>` in `params_json`.
- **FIFO eviction is opportunistic** — at each write, if `result_count > max_per_project` for the project, the oldest rows are deleted. No daemon required.
- **`query_text` is capped** — stored at `≤ truncate_query_text_bytes` bytes, truncated at a UTF-8 codepoint boundary (consistent with the 2 KiB source-snippet rule).

## Migration and upgrade notes

Migration `0005_provenance.sql` is applied automatically the first time any V0.3 command opens the store. It adds:

- `query_events` table + indexes
- `memory_events.memory_id` column + index (backfilled from `payload_json`)
- `memory_provenance` view (pre-joined for cheap `why` queries)

No manual step is needed. Existing `.vestige/config.toml` files round-trip without changes — the new `[traces]` section is optional and default-safe.

Existing V0.2 DBs are upgraded cleanly. If you downgrade to a V0.2 binary, the new tables and column remain but are ignored — no data loss.

## What is not in V0.3

- No GUI or TUI for trace browsing.
- No LLM-generated provenance narratives — the "why" walk is templated, not synthesised.
- No cross-project provenance — provenance and traces are project-scoped (waits for V0.9).
- No source diffing (was this snippet edited since capture?).
- No location fields on file/commit sources (`line_range`, `commit_sha`) — deferred to V0.4.
- No hard delete of traces — eviction is FIFO by configured cap only.
- No automatic dedup based on source overlap.
- No trace replay across embedding model versions without a warning — provider mismatch surfaces explicitly.

V0.4 is the **Memory browser (TUI)** milestone: an interactive `vestige browse` interface for list / expand / forget / restore, binding directly to the store without a daemon. The provenance walks and trace receipts from V0.3 feed directly into this — every memory you inspect in the browser will link to its provenance trail.
