# V0.4 — Memory Browser (TUI)

V0.4 is a single command: `vestige browse`. It opens a full-screen, keyboard-driven browser over the same data V0–V0.3 already produced — memories, candidates, traces, provenance receipts — and lets you list, inspect, mutate, and replay without leaving the terminal.

Nothing about the underlying store changes. There's no daemon. There's no new MCP tool. There's no schema change. V0.4 is a view layer.

If you already know V0.3 (`vestige why`, `vestige sources`, `vestige trace`), the browser is "the same data, navigable." If you're new, the browser is the fastest way to see what your project's memory looks like.

---

## TL;DR

```bash
cd ~/your-project
vestige init --name "My Project"   # if you haven't already
vestige browse
```

Three tabs: **Memories**, **Candidates**, **Traces**. `Tab` and `Shift-Tab` cycle. `j`/`k` move. `q` quits. `?` shows the keymap.

That's enough to get going. The rest of this doc walks the surface.

---

## Setup

`vestige browse` needs:

- a Vestige project (`vestige init` once per repo),
- a TTY on both stdin and stdout (it enters raw mode + the alternate screen).

If stdin or stdout is piped, the command fails fast with a friendly message pointing at the scriptable CLI commands (`list`, `show`, `search`, `why`, `sources`, `trace`).

```bash
vestige browse                  # default — opens on Memories
vestige browse --tab traces     # open directly on a specific tab
```

## The keymap

```
Navigation
  Tab / Shift-Tab   cycle tabs
  j / k or ↓ / ↑    move selection
  g / G             first / last
  Ctrl-d / Ctrl-u   half-page down / up
  /                 focus filter
  :                 command palette

Memory provenance (Memories tab)
  w                 why — provenance walk
  s                 sources — typed receipts
  t                 traces-of — which queries returned this

Mutations
  f                 forget memory (soft-delete, with confirm)
  r                 restore soft-deleted memory (with confirm)
  a                 approve candidate (with confirm)
  R                 reject candidate (reason prompt)
  p                 replay trace — diff against current store

Overlays
  ?                 toggle help
  Esc               close overlay / clear filter / back
  q / Ctrl-c        quit
```

`Esc` precedence: modal > palette > help > filter focus > sub-view. The same key always closes "the most-recently-opened thing."

## The Memories tab

The default view. Two-pane: list on the left (40%), detail on the right (60%).

- The list shows kind (decision / note / observation / preference / project_summary / open_question), then the title.
- Soft-deleted memories are listed with strike-through styling. They don't disappear — you can still inspect them and `r` to restore.
- The detail pane re-queries `get_memory` on every cursor move; for a local SQLite store this is sub-millisecond.
- The detail pane shows **every property** of the selected memory — id · type · importance · confidence · status · created/updated/deleted timestamps (absolute UTC + relative "3d ago") · title · every available representation (one-liner / summary / compressed / full, skipping duplicates) · full source list with kind / ref / 240-char preview / truncated flag. Press `w` / `s` / `t` to switch to the dedicated provenance walks.

### Filtering

Press `/`. Type. The list narrows via FTS5 lexical search (`search_memories`). Every keystroke triggers a fresh query — no debounce because everything is local.

```
/auth_      → matches memories whose representations match "auth"
Esc         → clears the filter, restores the full list
```

FTS5 operators pass through verbatim: `"phrase"`, `term1 AND term2`, `term1 OR term2`.

Soft-deleted memories are excluded from filtered results — `search_memories` is built on top of an FTS index that excludes deleted rows.

### Provenance sub-views

With a memory selected:

- `w` swaps the detail pane for a **why walk** — every journal event for this memory in chronological order, the same data as `vestige why <id>`.
- `s` swaps to the **sources view** — typed receipts (`file`, `commit`, `url`, `agent_session`, `mcp_call`, `candidate`, `manual`), same as `vestige sources <id>`.
- `t` swaps to **traces-of** — every recall query whose result set contains this memory. New in V0.4. The store query is a `LIKE` scan over `query_events.result_ids_json`, anchored on the full quoted ID so ULID substring matches don't false-positive.

Each sub-view shows a breadcrumb hint (`Esc — back to detail`). Sub-view data is loaded lazily on first request and cached until the cursor moves.

### Mutations

- `f` opens a confirm modal: `Forget memory mem_…?  y - yes  n / Enter / Esc - no`. Default is no — pressing Enter never triggers a destructive action.
- `r` opens the same modal but only when the selected row has `status = deleted`. On active memories it's a no-op (the modal never opens).
- After confirm, the list reloads, the cursor stays where it was, and the status row flashes the outcome (`Forgot mem_…` in green, or red on error).

Forget is soft-delete (PRD invariant) — no row is ever removed. `r` brings it back.

## The Candidates tab

Same shape as Memories. Default sort is most-recent-first. Each row shows kind, confidence, and title.

- `/` filters the visible list by case-insensitive substring on title or one-liner. Candidates are low-volume so we don't need FTS — substring is fine.
- `w` and `s` work identically to the Memories tab but read from the candidate journal (`fetch_candidate_events`, `fetch_candidate_sources_with_ids`).
- `a` approves the selected candidate. Confirmation modal, then `vestige_engine::approve_candidate` runs with default overrides. The new `mem_<ULID>` appears in the status flash.
- `R` (Shift+r) opens a **reject reason prompt** — a different modal shape from the binary confirm. It accepts free-form text:

```
Reject candidate
cand_01HX0000000000000000000ABC

Reason (empty = unspecified):
> stale_

Enter - submit     Esc - cancel
presets: duplicate | wrong | not_durable | too_noisy | stale
```

Empty input becomes `RejectionReason::Other("unspecified")`. Recognised tokens parse to the typed variant; anything else passes through as `Other(text)`.

After approval or rejection, the candidate drops from the pending inbox and the cursor moves to the next entry.

## The Traces tab

Read-only over `query_events`. Traces are append-only audit — you can't forget them, only inspect or replay.

- Each row shows kind (search / expand / context), caller (cli / mcp), mode (lexical / semantic / hybrid), result count, and a query preview.
- The detail pane shows the full query, parameters (pretty-printed JSON), provider/model when relevant, and the ordered result list with scores.

### Replay

Press `p` on a selected trace.

```
Detail · replay diff
replay of trace_01HX0000000000000000000ABC
new trace: trace_01HX0000000000000000000NEW

⚠  provider mismatch — replay re-ran with the current provider

added (1)
  + mem_01HX… (new memory not present at original recall)

removed (1)
  - mem_01HX… (forgotten since the original)

score changes (1)
  Δ +0.234  mem_01HX…  (memory still present, score improved)

corpus size: 47
```

What you're seeing: the same query re-run against the current store, with a structured diff against the original result set. Replay writes its own trace tagged with `replay_of` in `params_json` — so replays appear in the traces list themselves.

Two banners can appear at the top:

- **provider mismatch** — the original trace was generated with an embedding provider that differs from the one configured now (e.g. different dimensions or name). V0.4.x wires the configured `[embeddings]` provider into replay; if the names match, the banner is absent. Only truly mismatched providers (or unconfigured ones) show it.
- **mode fallback** — the original was `semantic` or `hybrid` but had to fall back to `lexical` because no provider was available. Replays remain useful; the diff just reflects the fallback.

`Esc` or any cursor movement returns the pane to the default detail view.

## The command palette (`:`)

Press `:`. The status line turns into a typed prompt. Useful when you know what you want and don't want to navigate to it.

| Command | What it does |
|---|---|
| `:quit` / `:q`              | Same as `q` |
| `:help`                     | Same as `?` |
| `:goto <id>`                | Cross-tab jump by `mem_…` / `cand_…` / `trace_…` ID. Switches tab automatically. |
| `:kind <type>` / `:kind all`| Memories tab — filter by memory type (decision / note / observation / preference / project_summary / open_question). |
| `:status active|deleted|all`| Memories tab — show only active, only deleted, or both. |
| `:caller cli|mcp|all`       | Traces tab — filter by caller. |
| `:search <text>`            | Equivalent to `/<text>` on the current tab (not yet on Traces). |
| `:mode lexical\|semantic\|hybrid` | Set the active search mode for the Memories `/` filter. Persists for the session. |

Unknown commands surface inline in red without dismissing the prompt — fix the typo and Enter again.

Status line shows live counts when not in palette mode:

```
Vestige · proj_my-project · [Mem 47 · Cand 3 · Trc 184]
```

When a non-default search mode is active:

```
Vestige · proj_my-project · [Mem 47 · Cand 3 · Trc 184] · mode:hybrid
```

When the configured provider was unavailable and the browser fell back:

```
Vestige · proj_my-project · [Mem 47 · Cand 3 · Trc 184] · mode:hybrid→lexical
```

When a mutation has just landed, the counts are replaced by a flash like `Approved cand_… → mem_…` until the next non-trivial input.

## Configuration

V0.4 needs no new config. It uses the same `.vestige/config.toml` every other command reads, and the same `~/.vestige/projects/<project-id>/memory.sqlite` store.

`NO_COLOR=1` is honoured: the active tab falls back to reverse-video styling, and no chromatic colours are emitted. Useful for screenshots and terminals without colour support.

---

## What's not in V0.4

The PRD §4 non-goals, in case they bite you:

- **No interactive memory capture.** Capture stays CLI/MCP. The browser is read+curate.
- **No memory body editing.** Memories are derived from the journal; editing them breaks the journal-is-truth invariant. To "fix" a memory: forget it and re-record.
- **No source location fields** (`line_range`, `commit_sha`, `byte_range`). Deferred to V0.5; the browser will pick them up when they ship.
- **No `--json` snapshot of the browser.** The CLI commands already expose programmatic access.
- **No per-user colour theme.** One fixed palette in V0.4 (plus `NO_COLOR`).
- **No mouse support.** Keyboard only.
- **No cross-project browse.** Strictly project-scoped, like every other surface.
- **Pre-warming semantic vectors at browser start.** Provider construction is lazy — the model loads on first semantic/hybrid use. Pre-warming stays a V0.5 / daemon concern.

## Architecture notes

For curious eyes:

- One ratatui `Terminal<CrosstermBackend<Stdout>>` for the session lifetime. No background threads. The poll cadence is 250ms — enough to keep `Ctrl-c` responsive without spinning a CPU core.
- Pure state in `crates/vestige-cli/src/commands/browse/app.rs` — `App`, per-tab states, `Action` enum, `Modal` (binary confirms and the text-input prompt), `CommandPalette`.
- Event mapping in `input.rs` — a single `map_event(event, &app)` function that switches behaviour by what's currently focused (modal, palette, filter, normal).
- I/O-bearing actions (cursor moves, filter edits, mutations, replay, palette commands) live in `mod.rs::apply_action`, which holds `&mut Store`. The pure-state module never sees the store.
- Snapshot tests over `ratatui::backend::TestBackend` cover every state shape — populated lists, empty states, filter prompts, sub-views, modals, replay diffs, status flash, palette errors.
- A panic hook restores the terminal before the default panic printer runs — a crash doesn't leave your shell in raw mode.

If you want to extend the browser, the tab modules under `crates/vestige-cli/src/commands/browse/tabs/` are the place to start. Each is roughly: `reload_list` + `refresh_detail` + `draw` + tests.
