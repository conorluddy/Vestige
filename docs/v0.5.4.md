# V0.5.4 — Autonomous session-log ingestion (daemon + CLI)

V0.5.3 shipped the **agent-driven** path: the `vestige_scan_sessions` MCP tool hands the
calling agent redacted transcript turns and the agent files candidates. V0.5.4 ships the
**autonomous** path deferred from it (epic #113): a configured LLM does the extraction with
no agent in the loop, driven either by the daemon on a schedule or by a one-shot CLI.

Both paths share the same source layer, redaction, cursor, and project-scope invariants —
and both route every result through the V0.2 inbox. Nothing is auto-promoted.

## The `vestige-extract` crate (#103)

A new internal crate mirroring `vestige-embed`:

- `ExtractionProvider` — a synchronous `Send + Sync` trait: `extract(&[NormalizedTurn]) ->
  Result<Vec<ExtractedCandidate>, ExtractError>`.
- Backends: `fake` (always compiled, deterministic — for tests), and feature-gated `ollama`
  (default real provider, no API key), `anthropic` (`ANTHROPIC_API_KEY`), and `openai`
  (`OPENAI_API_KEY`). Enable with `--features extract-ollama` / `extract-anthropic` /
  `extract-openai`.
- `build_provider(&ExtractionConfig)` factory; `ExtractError::ProviderDisabled(&'static str)`
  vs `UnknownProvider(String)`.

Configure it in `.vestige/config.toml`:

```toml
[extraction]                 # daemon + CLI only; never read by the MCP tool
provider = "ollama"          # "ollama" (default) | "anthropic" | "openai" | "fake"
model    = "llama3.2"        # optional; provider default otherwise
```

Dependency edge stays one-way: `vestige-engine → vestige-extract → vestige-core`
(`NormalizedTurn` moved into `vestige-core` so engine and extract share it).

## The shared scan core

`vestige_engine::scan_and_propose(sources, store, project_id, extractor, opts)` is the single
implementation both consumers call. Per in-scope session it:

1. reads transcript turns past the per-file watermark,
2. redacts secrets (secrets never reach the model),
3. hands a bounded batch to the `ExtractionProvider`,
4. maps each `ExtractedCandidate` to a candidate with `session_log` provenance and calls
   `propose_candidate`,
5. advances the watermark over what it read (skipped entirely in `--dry-run`).

A per-session extraction error is logged and that session is skipped **without** advancing
its cursor, so it is retried next scan — the provider never dumps raw turns as candidates.

## One-shot CLI (#107)

```bash
vestige scan --dry-run            # extract + preview; writes nothing, cursor unmoved
vestige scan                      # file candidates into the inbox via [extraction]
vestige scan --provider fake      # override the provider/model for this run
vestige inbox                     # review what landed
```

`--json` emits a structured report (`sessions_scanned`, `turns_processed`,
`candidates_proposed`, `cursor_advanced`, `previews[]`). With the default `ollama` provider
in a build without `--features extract-ollama`, the CLI errors with an actionable message;
use `--provider fake` for a dry test or rebuild with the feature.

## Daemon job (#106)

The daemon gains a `session_log_scan` scheduled job:

```toml
[daemon]
session_log_scan_interval_secs = 1800   # default 30 min; 0 disables the job
```

```bash
vestige daemon kick scan          # run a scan now across all supervised projects
vestige daemon status             # next_jobs[] includes session_log_scan when enabled
```

Each per-project worker builds that project's `[extraction]` provider from its config. When
the provider is unavailable (unconfigured, or feature-gated out of the daemon build), the
scan is a **no-op** logged at `warn` — passive ingestion stays strictly opt-in.

## Invariants

Unchanged: soft-delete only, project-scope boundary, immutable migrations, opt-in daemon,
and **no new write paths** — every candidate goes through the existing `propose_candidate`
inbox flow.
