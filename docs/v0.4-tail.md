# V0.4.1 — Tail tab

A fourth top-level tab on `vestige browse` for live monitoring of project memory activity. The `tail -f` equivalent for your repo's memory layer.

## What it is

```bash
vestige browse --tab tail
```

Opens the browser directly on the Tail tab. A merged, time-ordered stream of recent **memories** and pending **candidates** for the current project, ordered by `created_at` DESC. The view re-queries the store on a configurable interval (default 60s) and redraws in place.

If you've ever wanted to see the moment a new memory or candidate lands — while an agent is working, while you're capturing notes, while a heartbeat ingests — this is the surface for it.

## Why it exists

The other three browser tabs (Memories, Candidates, Traces) are interactive but static. They don't auto-refresh. The macOS menu-bar app (V0.5.1) shows steady-state counts but not the moment something appears. Tail closes that gap with the minimum surface that gets the job done: one new tab on a TUI that's already running.

## What you see

```
┌Tail (16)───────────────────────────────────────────────────────────────────────┐
│> obs   cand 0.50 second tail test candidate 1779569617                         │
│  note  mem  short note one-liner                                               │
│  note  cand 0.50 tail tab test candidate 1779569577                            │
│  note  mem  another note that was just captured                                │
│  dec   mem  SQLite stores durable memory; vector indexes remain replaceable    │
│  q     mem  Should we add a Linux systemd unit alongside the LaunchAgent?      │
│  …                                                                             │
└────────────────────────────────────────────────────────────────────────────────┘
 polling 60s · last poll 12s ago · depth:oneliner
```

Each row carries a kind tag (`note` / `dec` / `obs` / `q` / `pref` / `sum`), a source marker (`mem` or `cand`), an optional confidence score for candidates, and the body text rendered at the active disclosure depth.

## Controls

| Key                | Action                                                                |
|--------------------|-----------------------------------------------------------------------|
| `j` / `k`          | Move selection down / up.                                             |
| `g` / `G`          | Jump to top / bottom.                                                 |
| `Tab` / `Shift-Tab`| Cycle tabs.                                                           |
| `d`                | Cycle disclosure depth: one-liner → summary → compressed → one-liner. |
| `:`                | Open the command palette.                                             |
| `q`                | Quit.                                                                 |

Mutation keys (`f` / `r` / `a` / `R`) are inert on Tail — it's read-only. Switch to the Memories or Candidates tab to act on a row.

## Auto-scroll

Auto-scroll keeps the newest row at the top by default. The moment you press `j` (or any key that moves the cursor off row 0), auto-scroll **pauses** — the status bar appends `· paused`, new items continue to load on the next poll, but the cursor stays on the row you were looking at. Press `g` to return to the top and auto-scroll resumes.

This is the standard `tail -f` + pager UX: glance-friendly when idle, stable when you're inspecting something specific.

## Polling

Default: every 60 seconds. The browser ticks every 250ms regardless (for input responsiveness) and triggers a store re-query when `last_poll + interval` has elapsed.

Change it on the fly via the command palette:

```
:interval 5      # poll every 5 seconds (range: 5-3600)
:interval 300    # poll every 5 minutes
```

The polling cadence shows in the status bar.

## Capping the stream

Default: 200 most recent rows across memories + candidates combined. Adjust:

```
:tail 50         # cap at 50 rows
:tail 500        # cap at 500 rows
```

A `:tail` change triggers an immediate reload.

## Disclosure depth

Press `d` to cycle the rendered depth for memory rows:

- **one-liner** (default) — single-line `MemoryCard.one_liner` text. Densest view.
- **summary** — the summary representation if available, otherwise falls back to one-liner. Multi-line.
- **compressed** — the compressed representation if available, otherwise falls back to summary, then one-liner.

The active depth shows in the status bar: `depth:oneliner | depth:summary | depth:compressed`. Memories without richer representations stay at one-liner regardless of the setting.

Candidate rows are unaffected — they always render their title.

## Scope

Same `.vestige/config.toml` rule as every other view: **current project only**. The Tail tab will never show memories from a different project. Cross-project tail waits for V0.7 federation work.

## Verification

Two terminals against the repo (Vestige dogfoods itself):

```bash
# Terminal A
vestige browse --tab tail

# Terminal B
vestige note add "verifying tail tab"
vestige candidate add --type observation --body "test candidate"
```

Within 60s (or `:interval 5` immediately) both records appear at the top of Terminal A's stream, with the status bar resetting `last poll 0s ago`. Press `j` to move off the top → `· paused` appears; new records load but the cursor stays put. Press `g` to return → `paused` clears.

## Out of scope (follow-ups)

- **Cross-project tail** — waits for V0.7 federation.
- **Mutations from the Tail tab** (`f` / `a` / `r` / `R`) — read-only by design; switch tabs.
- **Composite index on `memories(project_id, status, created_at DESC)`** — only added if profiling shows the unindexed sort matters at realistic N.
- **Tail data in the Swift menu-bar app** — separate plan; this tab is the CLI half.
